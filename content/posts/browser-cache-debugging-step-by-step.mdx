“배포했는데 왜 화면이 그대로지?”라는 질문은 대부분 캐시 문제로 연결됩니다.  
하지만 캐시 레이어를 섞어서 보면 원인을 더 늦게 찾습니다.

## 캐시를 레이어로 나눠 본다

문제는 보통 아래 3층 중 하나입니다.

- 브라우저 캐시
- CDN/엣지 캐시
- 앱 내부 정적 자산 참조

레이어를 분리하지 않으면 디버깅이 감으로 흐릅니다.

## 1단계: 브라우저 확인

- 시크릿 창에서 재현 확인
- 강력 새로고침으로 비교
- Network 탭에서 응답 헤더 확인 (`cache-control`, `etag`)

시크릿에서도 동일하면 브라우저 로컬 캐시 가능성은 낮습니다.

## 2단계: CDN/엣지 확인

- 배포 URL과 정적 자산 URL을 분리 확인
- 동일 자산을 쿼리스트링 버전으로 호출해 비교
- `x-cache` 계열 헤더로 히트/미스 확인

CDN이 오래된 자산을 유지하면 앱 코드는 정상이어도 화면이 안 바뀝니다.

## 3단계: 앱 레퍼런스 확인

- HTML이 새 빌드 자산을 가리키는지
- chunk 파일명이 실제 배포와 일치하는지
- 서비스워커가 구버전을 잡고 있지 않은지

특히 PWA 설정이 있으면 서비스워커 캐시가 핵심 원인일 수 있습니다.

## 대응 우선순위

1. 사용자 영향 큰 페이지부터 우회 경로 제공
2. 자산 버전 강제 갱신
3. CDN 캐시 무효화
4. 재발 방지로 배포 체크리스트 업데이트

복구와 원인분석을 동시에 하려 하면 둘 다 느려집니다.

## 재발 방지 규칙

- 정적 자산 파일명에 해시 고정
- 캐시 정책 문서화
- 배포 후 시크릿 창 스모크 1회 의무화

작은 규칙 3개만 지켜도 “반영 안 됨” 이슈가 크게 줄어듭니다.

## 체크리스트

- [ ] 시크릿/일반 모드 재현 차이를 확인했는가
- [ ] CDN 히트/미스 헤더를 확인했는가
- [ ] HTML이 최신 자산을 참조하는가
- [ ] 서비스워커 개입 여부를 점검했는가
