AI에게 “알아서 해줘”라고 말하면, 가끔은 빠르게 끝나고 가끔은 길게 돌아갑니다.  
차이는 모델 능력보다 **요청의 구조**에서 자주 갈립니다.

저는 블로그를 다듬을 때 아래 4줄을 기본 템플릿으로 씁니다.

## 1) 목표를 1문장으로 고정

먼저 “왜 하는지”를 짧게 고정합니다.

- 예: 메인 최신 글 벤토 레이아웃의 빈 공간을 없앤다
- 예: 파비콘/OG/Schema를 정리해 검색/공유 품질을 올린다

목표가 흔들리면, 구현은 빨라도 결과가 분산됩니다.

## 2) In/Out 범위를 먼저 적는다

모호한 요청의 대부분은 “하지 말아야 할 것”이 비어 있어서 생깁니다.

- In: 수정할 컴포넌트, 페이지, 검증 범위
- Out: 이번 턴에서 건드리지 않을 범위

이걸 쓰면 AI가 불필요한 확장 작업을 덜 합니다.

## 3) 성공 기준을 숫자로 둔다

“좋아 보이게” 대신 숫자/조건으로 씁니다.

- lint/build 통과
- 데스크톱 1280 기준 카드 하단 정렬
- 콘솔 에러 0
- 이미지 alt 누락 0

요청의 품질은 “좋은 문장”보다 “측정 가능한 조건”에서 올라갑니다.

## 4) 마지막에 배포 조건을 명시

실무에서는 구현보다 배포 단계에서 자주 꼬입니다.

- 브랜치: `main` 직접 반영인지, PR인지
- 배포: Vercel 자동 배포 확인 여부
- 보고: 변경 파일/검증 결과/링크 전달 형식

여기까지 미리 쓰면 턴이 끝나고 다시 왕복할 일이 줄어듭니다.

## 바로 쓰는 템플릿

```md
목표:
- (한 줄)

Scope:
- In: (수정 파일/영역)
- Out: (이번에 제외)

완료 기준:
- (정량 조건 3~5개)

검증:
- npm run lint
- npm run build
- (필요 시) Playwright 스모크

배포:
- main 푸시 + 배포 링크 공유
```

## 오늘의 메모

요청을 잘 쓰는 건 친절함이 아니라 **비용 절감**에 가깝습니다.  
한 번 명확하게 쓰면, 수정 횟수와 피로가 같이 줄어듭니다.
