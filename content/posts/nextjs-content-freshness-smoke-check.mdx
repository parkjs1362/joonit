글을 발행해도 사용자 화면에 늦게 반영되면 운영 신뢰가 떨어집니다.  
그래서 배포 성공 여부와 **콘텐츠 최신성**을 분리해서 확인해야 합니다.

## 1) 검증 URL 3종을 먼저 고정한다

배포 후 매번 다른 URL을 확인하면 누락이 생깁니다.

- 목록 페이지(`/blog`)
- 신규 글 상세 페이지(`/blog/[slug]`)
- 피드(`/feed.xml`)

이 3개만 고정 확인해도 대부분의 반영 지연을 조기에 잡을 수 있습니다.

## 2) 확인 지표를 시간 기준으로 단순화한다

운영에서 중요한 건 완벽한 분석보다 "몇 분 안에 반영됐는가"입니다.

- 배포 후 3분: 상세 페이지 노출 확인
- 배포 후 5분: 목록 정렬/카테고리 반영 확인
- 배포 후 10분: 피드 갱신 확인

시간 기준을 두면 원인 분석 전에도 즉시 대응 판단이 가능합니다.

## 3) 실패 시 대응 순서를 표준화한다

문제 발생 시 임의 대응을 줄이기 위해 순서를 고정합니다.

1. slug·date·frontmatter 누락 여부 확인
2. 빌드 결과와 런타임 캐시 경로 확인
3. 재배포 전 최소 변경으로 재검증

고정 순서가 있으면 재현성 없는 핫픽스를 줄일 수 있습니다.

## 체크리스트

- [ ] 고정 검증 URL 3종을 모두 확인했는가
- [ ] 3/5/10분 기준 검증 시간을 기록했는가
- [ ] 실패 대응 순서대로 원인 점검을 했는가
- [ ] 임시 수정 없이 재현 가능한 해결 절차를 남겼는가

## Joonit의 관점

콘텐츠 운영에서 "배포 성공"은 절반입니다.  
나머지 절반은 사용자가 보는 시점의 최신성을 정해진 시간 안에 보장하는 것입니다.
