접근성 경고는 보통 “나중에”로 밀리기 쉽습니다.  
하지만 운영 사이트에서는 가장 빨리 신뢰를 깎는 항목이기도 합니다.

저는 접근성 이슈를 아래처럼 기능 개발과 같은 루프로 처리합니다.

## 1) 경고를 문장 그대로 기록

먼저 경고를 해석하지 말고 그대로 적습니다.

- `Images must have alternate text`
- `Elements must meet minimum color contrast ratio thresholds`

문장을 그대로 보관하면, 원인 추적이 훨씬 빨라집니다.

## 2) 컴포넌트 단위로 원인 분리

대부분 이슈는 페이지가 아니라 컴포넌트에서 반복됩니다.

- 카드 이미지 컴포넌트 alt 누락
- 공통 버튼/링크 색상 대비 부족
- 다크모드에서만 대비 무너짐

한 번 고친 뒤 전체에 반영하려면 공통 컴포넌트가 기준점이어야 합니다.

## 3) 수정 기준을 숫자로 둔다

주관적 판단을 줄이기 위해 기준을 명시합니다.

- `img:not([alt])` 0개
- 의미 없는 장식 이미지는 의도적으로 빈 alt
- 텍스트/배경 대비가 라이트/다크 모두에서 읽히는지 확인

특히 다크모드는 “괜찮아 보인다”가 자주 틀립니다.

## 4) 회귀 방지

수정 후 한 번 더 깨지는 이유는 보통 신규 콘텐츠입니다.

- 새 글 추가 시 커버 alt 자동 규칙 유지
- 카테고리 카드/상세 카드 모두 동일 컴포넌트 사용
- 배포 전 최소 스모크 QA(홈/블로그/상세)

## 5) 운영 체크

- 린트/빌드 통과
- 콘솔 에러/접근성 경고 재확인
- 주요 페이지에서 키보드 포커스 이동 확인

## 오늘의 메모

접근성은 “좋으면 좋은 것”이 아니라  
사이트의 기본 완성도를 증명하는 가장 빠른 신호입니다.
