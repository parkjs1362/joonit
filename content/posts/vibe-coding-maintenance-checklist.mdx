바이브 코딩은 속도가 장점입니다.  
하지만 속도가 빨라질수록, 흔히 이런 문제가 같이 옵니다.

- 더미 링크/더미 문구가 남는다
- 접근성(alt/대비)이 깨진다
- 소개/정책 페이지가 빈약하다
- 배포는 되는데 "신뢰감"이 떨어진다

그래서 저는 아래 체크리스트를 운영 기본값으로 둡니다.

## 1) 더미 제거(신뢰감의 1순위)

- "Lorem ipsum" 같은 더미 문구
- LinkedIn 등 미사용 링크
- 실제로 존재하지 않는 메뉴/페이지

더미는 기능 버그보다 신뢰를 더 빨리 깎습니다.

## 2) 기본 페이지는 반드시 갖추기

- About(누가 쓰는지)
- Contact(연락 수단)
- Privacy/Terms(정책 문서)

콘텐츠가 좋아도, 기본 문서가 없으면 사이트가 "미완성"으로 보입니다.

## 3) 배포 전 3개 커맨드만은 자동으로

- lint
- build
- (가능하면) 간단한 접근성 점검

여기서 한 번 걸러지면, 운영 중 사고가 크게 줄어듭니다.

## 4) 디자인은 '효과'보다 '지속'이 기준

멋있는 인터랙션보다 중요한 건,

- 오래 읽기 편한 본문
- 일관된 카드/커버
- 깨지지 않는 대비와 여백

이게 쌓이면, 바이브로 만든 프로젝트도 "완성된 제품"처럼 보입니다.

## 오늘의 메모

바이브 코딩의 핵심은 "빠르게 만든다"가 아니라,

> 빠르게 만든 뒤, 반복적으로 다듬을 수 있게 만드는 것

체크리스트는 창의력을 죽이는 게 아니라, 반복 피로를 줄여줍니다.

## 근거: 바이브 코딩 이후 유지보수가 핵심인 이유

빠른 생성은 시작점일 뿐이고, 신뢰는 운영 단계에서 결정됩니다.

- **운영 부채 누적**: 더미 정보, 깨진 링크, 누락 문서가 신뢰를 빠르게 깎습니다.
- **배포 안전성 필요**: 작은 변경도 검증 없이 반영하면 회귀가 누적됩니다.
- **시각 일관성 효과**: 기능보다 일관된 UI 규칙이 품질 인상을 좌우합니다.
- **체크리스트 외부화**: 기억에 의존하지 않고 절차로 고정해야 반복 품질이 유지됩니다.

### 측정 지표

- 배포 후 회귀 건수
- 깨진 링크/더미 텍스트 탐지 건수
- 접근성 경고 수
- 수정 후 재배포까지 걸린 시간

## 실행 방법: 유지보수 체크 프로토콜

1. **신뢰 항목 우선 점검**: 연락처/소개/정책/저작권 정보를 먼저 확인합니다.
2. **기술 점검 3종 고정**: lint, build, 핵심 경로 스모크 테스트를 항상 수행합니다.
3. **시각 점검 기준화**: 카드 높이, 텍스트 대비, 모바일 오버플로우를 체크합니다.
4. **변경 로그 기록**: 무엇을 바꿨는지와 검증 결과를 함께 남깁니다.
5. **주간 정기 점검**: 기능 추가보다 안정성 점검 슬롯을 먼저 확보합니다.

### 실패 대응

- 회귀가 반복되면 기능 추가를 중단하고 체크리스트 미이행 항목부터 복구합니다.
