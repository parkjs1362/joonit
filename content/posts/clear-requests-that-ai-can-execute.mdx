AI에게 요청할 때 가장 흔한 문제는  
정보 부족보다 **추상어 과다**입니다.

예를 들어 “트렌디하게 바꿔줘”는 좋은 시작이지만,  
실행 기준이 없어서 결과가 크게 흔들릴 수 있습니다.

## 1) 추상어를 측정어로 바꾼다

아래처럼 바꾸면 실행 가능성이 올라갑니다.

- 트렌디하게 → 카드 간격 24px, 타이포 계층 3단, 대비 기준 유지
- 많이 보여줘 → 최신 글 6개 고정
- 정리해줘 → 핵심 3개 섹션 + 체크리스트

핵심은 “보기 좋은 문장”이 아니라 “판단 가능한 조건”입니다.

## 2) 파일/경로를 명시한다

요청이 코드 변경이라면 대상 파일을 바로 적습니다.

- `app/page.tsx`
- `components/PostVisualCard.tsx`
- `app/layout.tsx`

경로가 있으면 AI가 탐색 시간을 줄이고 바로 수정에 들어갑니다.

## 3) 유지/삭제를 동시에 적는다

결과가 과해지는 이유는 “유지할 것”이 빠져 있기 때문입니다.

- 유지: 현재 톤, 카테고리 구조, 모바일 레이아웃
- 변경: 데스크톱 카드 높이 정렬
- 삭제: 더미 문구/사용 안 하는 링크

이렇게 쓰면 불필요한 리라이트가 줄어듭니다.

## 4) 마지막 한 줄: 검증 기준

요청 끝에 검증을 붙입니다.

- lint/build 통과
- 주요 페이지 스모크 확인
- 콘솔 에러 0

검증 기준이 없으면 “수정 완료”의 기준도 모호해집니다.

## 바로 쓰는 문장 패턴

```md
목표: (한 줄)
유지: (건드리지 않을 것)
변경: (구체 수정 2~4개)
완료 기준: (정량 2~3개)
```

## 오늘의 메모

좋은 요청은 길어서가 아니라  
판단 기준이 명확해서 결과 편차를 줄입니다.

## 근거: 명확한 요청이 결과 품질을 바꾸는 이유

AI 협업의 품질은 모델보다 입력 명세의 품질에 크게 좌우됩니다.

- **모호성 비용**: 추상 요청은 재질문과 재작업을 증가시킵니다.
- **경계 조건 필요**: 유지/삭제 범위를 지정하지 않으면 원치 않는 수정이 섞입니다.
- **검증 가능성**: 완료 기준이 명확해야 결과를 빠르게 판단할 수 있습니다.
- **경로/파일 명시 효과**: 작업 단위를 구체화하면 실행 시간이 줄고 누락이 감소합니다.

### 측정 지표

- 1회 요청당 재작업 횟수
- 요청 후 추가 질문 수
- 첫 결과 채택률
- 완료까지 총 소요 시간

## 실행 방법: 요청 명세 템플릿

1. **목표 한 줄**: 무엇을 바꾸고 왜 바꾸는지 한 문장으로 씁니다.
2. **범위 지정**: 수정 파일/수정 금지 파일을 분리해 명시합니다.
3. **행동 단위 정의**: “개선” 대신 “폰트 교체, 타이포 축소, 배지 제거”처럼 동사 단위로 씁니다.
4. **검증 기준 추가**: lint/build/화면 확인 기준을 숫자로 씁니다.
5. **완료 조건 고정**: 최종 상태를 체크리스트로 닫습니다.

### 바로 쓰는 DoD 예시

- lint/build 통과
- 특정 문구 0건
- 모바일 가로 스크롤 0
- 지정 파일 외 변경 없음
