AI 협업에서 같은 요청을 반복했는데 결과가 달라지면, 모델 문제로만 보기 쉽습니다.  
실제로는 프롬프트가 구두 지시처럼 흩어져 있어서 생기는 경우가 많습니다.

그래서 프롬프트도 코드처럼 버전관리해야 합니다.

## 왜 버전관리가 필요한가

프롬프트가 문서 채팅에 흩어져 있으면 아래 문제가 생깁니다.

- 어떤 문구가 효과 있었는지 추적 불가
- 나쁜 수정이 들어와도 롤백 어려움
- 팀원이 기준을 공유하지 못함

버전관리의 목적은 멋진 체계가 아니라 **재현성 확보**입니다.

## 최소 운영 단위: Prompt Spec 1장

프롬프트 하나당 아래 항목을 고정합니다.

- 목적(무엇을 만들려는가)
- 입력 형식(필수/옵션)
- 출력 형식(길이/구조)
- 금지 조건(민감정보, 과장표현 등)
- 검증 기준(완료 조건)

이 5개가 있으면 품질 대화가 쉬워집니다.

## 변경 로그 템플릿

프롬프트 수정 시 아래 3줄만 남겨도 충분합니다.

- 변경 이유: 왜 바꿨는가
- 기대 효과: 무엇이 좋아져야 하는가
- 회귀 위험: 무엇이 나빠질 수 있는가

짧아도 축적되면 팀의 운영 지식이 됩니다.

## 회귀 테스트는 3케이스면 시작 가능

대규모 세트가 없어도 됩니다.  
대표 입력 3개만 고정해서 비교합니다.

- 정상 케이스
- 경계 케이스(입력 부족)
- 금지 케이스(민감정보 포함)

버전 업데이트마다 세 케이스 결과를 비교하면 품질 편차를 빠르게 잡을 수 있습니다.

## 실패 줄이는 규칙

- 프롬프트 수정은 하루 1회 배치로 묶기
- 실험 프롬프트와 운영 프롬프트 분리
- 운영 프롬프트는 승인자 1명 지정

규칙이 있어야 “즉흥 수정”이 줄어듭니다.

## 체크리스트

- [ ] 변경 로그에 이유/기대/리스크가 있는가
- [ ] 대표 입력 3케이스 회귀 결과를 남겼는가
- [ ] 운영 프롬프트와 실험 프롬프트를 분리했는가
