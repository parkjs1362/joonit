API 운영에서 흔한 실패는 “모든 오류를 다 잡으려는 태도”입니다.  
리소스가 제한된 서비스에서는 **허용 가능한 실패 범위**를 먼저 정해야 우선순위가 생깁니다.

## 에러 버짓이 필요한 이유

에러 버짓은 “서비스가 감당 가능한 실패량”입니다.  
이 기준이 없으면 다음 문제가 반복됩니다.

- 사소한 오류에 과도한 대응
- 핵심 장애 대응 지연
- 기능 개발과 안정화의 충돌

버짓은 개발 속도와 안정성의 균형점을 만드는 도구입니다.

## 최소 SLO 설정 방법

소규모 API라면 복잡한 지표보다 아래 2개부터 시작하는 것이 좋습니다.

- 성공률 SLO (예: 99.5%)
- p95 응답시간 SLO (예: 500ms)

핵심은 수치 그 자체보다 **모두가 같은 기준으로 판단하는 것**입니다.

## 에러 예산 계산 예시

월 100,000 요청에서 성공률 SLO가 99.5%라면 허용 실패는 500건입니다.

- 월 초반에 실패량을 빠르게 소진하면 기능 배포를 줄이고 안정화 작업 우선
- 버짓이 충분하면 계획된 기능 배포 유지

이 규칙만 있어도 회의가 길어지지 않습니다.

## 운영 절차

### 1) 오류를 분류한다

- 사용자 체감 오류(5xx, 타임아웃)
- 비핵심 내부 오류(재시도 성공)
- 관찰성 오류(로그 누락/메트릭 누락)

### 2) 주간 리뷰에서 버짓 소진율을 본다

- 25% 이하: 계획 유지
- 25~60%: 위험 구간, 배포 속도 조절
- 60% 이상: 안정화 스프린트 전환

### 3) 재발 오류를 우선 제거한다

단발성보다 반복 오류 제거가 장기 품질에 더 큰 효과를 냅니다.

## 지표를 운영 문장으로 바꾸기

데이터만 봐서는 행동이 나오지 않습니다.  
지표는 항상 행동 규칙과 연결해야 합니다.

- 오류율 급증 → 신규 기능 롤아웃 중지
- 특정 엔드포인트 p95 악화 → 캐시/쿼리 우선 점검
- 타임아웃 증가 → 재시도 정책/서킷브레이커 점검

## 체크리스트

- [ ] 월간 허용 실패량(에러 버짓)을 계산했는가
- [ ] 버짓 소진율에 따른 행동 규칙이 있는가
- [ ] 반복 오류 Top 3를 매주 추적하는가
- [ ] 지표와 배포 의사결정이 연결돼 있는가
