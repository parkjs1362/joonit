요청-수정-재요청이 반복되면,  
어느 순간 “지금 왜 이걸 고치고 있지?”라는 질문이 생깁니다.

저는 이 상황을 줄이기 위해 피드백 루프를 3단계로 고정합니다.

## 1) 변경 목적을 매 턴에 다시 붙인다

수정 요청이 짧아질수록, 목적 문장 하나를 매번 붙입니다.

- 예: 메인 가독성 개선이 목적
- 예: 애드센스 심사 통과 가능성 개선이 목적
- 예: 역사 글 신뢰도 강화가 목적

이 한 줄이 없으면 작업은 빨라도 방향이 흔들립니다.

## 2) “완료된 것”과 “다음 것”을 분리

여러 요청이 섞이면 체감상 아무것도 끝나지 않은 느낌이 듭니다.

그래서 매 턴 이렇게 분리합니다.

- 완료: 무엇을 바꿨는지
- 다음: 이번 턴에서 다룰 1~2개

완료와 다음이 분리되면, 피로가 확 줄어듭니다.

## 3) 단어를 통일한다

같은 문제를 서로 다른 단어로 부르면 추적이 어려워집니다.

- “레이아웃 깨짐” = 빈 공간/오버플로우/정렬 불일치 중 무엇인지
- “SEO” = 메타/스키마/OG 중 어디인지
- “접근성” = alt/대비/포커스 중 무엇인지

문제 이름이 통일되면 해결 속도도 빨라집니다.

## 4) 로그를 남기는 기준

모든 걸 길게 쓰기보다, 다음 4가지만 남깁니다.

- 수정 파일
- 검증 명령
- 배포 여부
- 남은 리스크

이 4개만 있어도, 다음 날 이어서 작업하기가 쉬워집니다.

## 오늘의 메모

반복 작업에서 가장 먼저 사라지는 건 시간보다 맥락입니다.  
맥락만 지키면 속도는 나중에 따라옵니다.

## 근거: 컨텍스트 손실이 생산성을 깎는 방식

피드백이 빠르게 오가도 맥락이 사라지면 같은 설명을 반복하게 됩니다.

- **재설명 비용**: 변경 목적과 제약이 누락되면 매 턴마다 다시 설명해야 합니다.
- **용어 불일치**: 같은 대상을 다른 이름으로 부르면 결론이 흔들립니다.
- **완료/다음 혼재**: 이미 끝난 항목과 다음 작업이 섞이면 우선순위가 흐려집니다.
- **기록 부재**: 왜 바꿨는지 남지 않으면 회귀가 반복됩니다.

### 측정 지표

- 턴당 재설명 횟수
- 변경 요청 대비 적용 정확도
- 같은 이슈 재발 빈도
- 의사결정 기록 누락 건수

## 실행 방법: 컨텍스트 보존 프로토콜

1. **턴 시작 3줄**: 목표, 제약, 이번 턴 완료 기준을 고정합니다.
2. **용어 사전 고정**: 핵심 대상 이름(컴포넌트/파일/규칙)을 통일합니다.
3. **결과 분리 기록**: 완료 항목과 다음 항목을 항상 분리해 기록합니다.
4. **결정 로그 1줄**: 왜 이 선택을 했는지 이유를 남깁니다.
5. **주간 정리**: 반복 재설명 이슈를 찾아 템플릿에 반영합니다.

### 실패 대응

- 재설명이 2턴 연속 발생하면 템플릿의 목표/범위 문장을 먼저 고칩니다.
