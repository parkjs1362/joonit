개인 블로그는 보통 빠르게 고치고 바로 배포합니다.  
문제는 이 속도가 누적되면 “작은 실수”도 잦아진다는 점입니다.

저는 main 직접 배포를 할 때 아래 순서를 고정합니다.

## 1) 수정 범위를 짧게 묶는다

한 커밋에 너무 많은 변경을 넣지 않습니다.

- 레이아웃 수정
- 콘텐츠 추가
- 메타/SEO 수정

이 셋을 한 번에 섞으면, 롤백 판단이 어려워집니다.

## 2) 배포 전 필수 2개

- `npm run lint`
- `npm run build`

이 두 개를 건너뛰면 배포 후 디버깅 시간이 길어집니다.

## 3) 스모크 QA는 페이지 4개만

매번 전체 QA를 하긴 어렵기 때문에 핵심만 고정합니다.

- `/`
- `/blog`
- `/blog/<최신글>`
- `/about` 또는 `/contact`

이 4개가 정상이면, 대부분의 치명 이슈를 초기에 잡을 수 있습니다.

## 4) 배포 후 확인 포인트

Vercel 배포가 “성공”이어도 끝이 아닙니다.

- 실제 사이트에서 캐시 이슈(아이콘/OG) 확인
- 콘솔 에러 확인
- 링크 404 확인

특히 파비콘/OG는 캐시 때문에 “코드는 맞는데 화면이 다르게 보이는” 일이 자주 생깁니다.

## 5) 리포트 포맷을 고정

배포 후 보고는 짧고 일정하게 작성합니다.

- 변경 파일
- 검증 결과(lint/build/스모크)
- 배포 URL
- 남은 리스크(있다면 1~2줄)

이렇게 하면 다음 작업자가 바로 이어받을 수 있습니다.

## 오늘의 메모

배포는 이벤트가 아니라 루틴입니다.  
루틴이 안정되면, 속도도 결국 같이 올라갑니다.
