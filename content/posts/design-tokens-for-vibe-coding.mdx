저는 “기술 스택”을 앞세우는 것보다, **읽히는 화면**을 만드는 쪽을 더 중요하게 봅니다. 특히 바이브 코딩처럼 빠르게 만들 때는요.

그럴 때 도움이 되는 최소 단위가 **디자인 토큰 6개**입니다.

## 토큰 6개만 정하자

이 6개만 고정해도 화면이 한결 안정적으로 보입니다.

- `background`: 페이지 배경
- `foreground`: 기본 텍스트
- `primary`: 버튼/링크/포인트
- `muted`: 보조 텍스트
- `border`: 경계선
- `card`: 카드/컨테이너 배경

예시:

```css
:root {
  --background: #fcfcfd;
  --foreground: #0b0d12;
  --primary: #2563eb;
  --muted: #667085;
  --border: #e6e8ee;
  --card: #ffffff;
}
```

## 토큰을 “클래스”로 바로 쓰게 만들기

토큰을 정해도, 매번 `var(--primary)`를 직접 쓰면 금방 흐트러집니다.

그래서 저는 토큰을 “내가 자주 쓰는 클래스”로 연결해둡니다. Tailwind v4라면 이런 식이죠.

```css
@theme inline {
  --color-background: var(--background);
  --color-foreground: var(--foreground);
  --color-primary: var(--primary);
  --color-muted: var(--muted);
  --color-border: var(--border);
  --color-card: var(--card);
}
```

이렇게 해두면 `bg-background`, `text-muted`, `border-border` 같은 표현으로 화면을 만들 수 있고,
“의미 기반”으로 스타일이 쌓이기 시작합니다.

## 토큰이 있으면 선택 피로가 줄어든다

기능을 만드는 중간중간 이런 고민을 하게 됩니다.

- “이 텍스트는 회색 몇 번?”
- “카드 배경은 흰색? 약간 회색?”
- “버튼은 파란색? 보라색?”

이게 쌓이면 레이아웃보다 **결정 피로**가 먼저 옵니다. 토큰은 그 피로를 줄여주는 기본값입니다.

## 가독성은 숫자로 고정해두는 게 편하다

디자인이 갑자기 좋아 보이는 순간은 대부분 타이포그래피에서 나옵니다.

- 본문 `line-height`: 1.7 ~ 1.9
- 본문 폭: 680 ~ 760px(대략 70~75ch)
- 모바일 기본 글자 크기: 너무 작지 않게

“내가 오래 읽어도 덜 피곤한가?”가 기준입니다.

## 컬러 사용 규칙(최소한만)

토큰이 있어도 컬러를 남발하면 신뢰감이 떨어집니다. 저는 룰을 이렇게 둡니다.

- `primary`는 **링크/버튼/강조**에만
- 일반 텍스트는 `foreground`와 `muted`만으로 해결
- 구분선/테두리는 `border`로만 통일

정리하면 “컬러로 디자인하지 말고, 레이아웃과 타이포로 디자인”하는 쪽이 결과가 좋았습니다.

## 토큰이 깨지고 있다는 신호

프로젝트가 커질수록 이런 현상이 보이기 시작합니다.

- 회색이 4~5종류로 늘어난다
- 카드 배경이 페이지마다 다르다
- 버튼이 화면마다 다른 느낌이다
- 다크모드에서 어떤 텍스트는 읽히고 어떤 텍스트는 안 읽힌다

이때는 새로운 컴포넌트를 만드는 것보다, **토큰을 다시 정리하는 게 더 빠른 개선**인 경우가 많습니다.

## 결론: 토큰은 디자인 시스템의 ‘씨앗’

거창한 디자인 시스템이 없어도 됩니다.

일단 토큰 6개를 정해두면,

- 화면이 일관돼 보이고
- 새 컴포넌트를 만들 때 고민이 줄고
- 결과물이 더 “신뢰감” 있게 느껴집니다.

바이브 코딩은 속도가 장점이니까, **기본값을 정해서 속도를 지키는 것**이 오히려 최고의 품질 관리가 됩니다.
