메뉴바 앱에서 가장 빨리 신뢰를 잃는 부분은 토글입니다.  
켜짐으로 보이는데 실제로는 안 돌아가거나, 꺼짐처럼 보이는데 백그라운드에서 살아 있는 경우가 생각보다 자주 생깁니다.

이런 앱은 기능 추가보다 먼저 **상태 표현을 믿을 수 있게 만드는 것**이 필요합니다.

## 1) 토글 상태는 “마지막 클릭 결과”가 아니라 “시스템 현재 상태”를 보여줘야 한다

많은 메뉴바 앱이 버튼을 누른 직후의 성공 메시지를 기준으로 상태를 바꿉니다.  
하지만 자동실행 같은 기능은 실제 시스템 등록 여부를 확인하기 전까지는 성공이 아닙니다.

그래서 토글 상태는 아래 순서가 맞습니다.

1. 사용자 클릭
2. 시스템 작업 실행
3. 시스템 상태 재조회
4. 조회 결과로 UI 갱신

이 순서를 지키면 느려도 신뢰가 쌓입니다.  
반대로 2단계까지만 하고 UI를 바꾸면 “빠르지만 틀린 토글”이 됩니다.

## 2) LaunchAgent 계층에서는 plist 존재 여부만 보면 부족하다

파일이 있다고 실행 중이라는 뜻은 아닙니다.  
반대로 파일이 없어도 이미 로드된 상태가 남아 있을 수 있습니다.

그래서 최소 세 가지를 나눠서 봐야 합니다.

- plist 파일 존재
- launchctl 등록 상태
- 최근 로그/실행 흔적

이 세 축이 맞아야 “켜짐”이라고 볼 수 있습니다.

## 3) 토글 실패는 조용히 묻히지 말고 원인별로 분기해야 한다

자동실행 토글이 실패했을 때 “실패했습니다” 한 줄만 띄우면 사용자는 다음 행동을 못 합니다.  
실패는 최소한 다음 정도로 나눠서 보여주는 편이 낫습니다.

- 권한/등록 실패
- 경로 문제
- 런타임 실행 실패
- 이미 로드된 상태 충돌

원인을 분리하면 사용자는 로그를 볼지, 다시 시도할지, 설정을 바꿀지 판단할 수 있습니다.

## 4) 토글에는 “확인 가능한 후속 지표”가 붙어야 한다

상태를 보강하는 가장 쉬운 방법은 마지막 성공 시각이나 오늘 실행 횟수 같은 보조 지표를 붙이는 것입니다.  
이 정보가 있으면 토글은 단순 스위치가 아니라 **운영 패널**에 가까워집니다.

예를 들어 아래 두 문장은 체감이 다릅니다.

- 자동실행: 켜짐
- 자동실행: 켜짐 / 마지막 실행 09:00 / 오늘 1회

후자가 훨씬 믿을 만합니다.

## 5) 메뉴바 앱 디버깅은 앱 안보다 바깥 증거를 더 많이 봐야 한다

메뉴바 앱 안에서 상태를 잘 보여줘도, 실제 운영 이슈는 앱 바깥에서 드러나는 경우가 많습니다.

- `launchctl list` 결과
- 로그 파일 갱신 시간
- 실제 작업 결과물 생성 여부

이런 바깥 증거가 있어야 “UI는 켜져 있는데 왜 작동 안 하지?” 같은 상황을 빠르게 좁힐 수 있습니다.

## 체크리스트

- [ ] 토글 상태가 마지막 클릭 결과가 아니라 시스템 재조회 결과를 반영하는가
- [ ] plist, 등록 상태, 로그를 분리해서 확인하는가
- [ ] 실패 원인이 분류돼 보이는가
- [ ] 마지막 실행 시각 같은 보조 지표가 있는가
- [ ] 앱 바깥 증거로도 상태를 검증하는가
