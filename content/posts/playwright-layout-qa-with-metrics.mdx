디자인 수정 후에 가장 자주 나오는 말은  
“뭔가 아직 어색해요”입니다.

이 문장을 줄이려면, 시각 검토에 **수치 기준**을 추가해야 합니다.

## 1) 먼저 측정할 지표를 1~2개로 고정

저는 레이아웃 이슈를 아래처럼 단순화합니다.

- `blankGap`: 컨테이너 하단과 카드 하단의 차이
- `deltaBottom`: 좌/우 카드 하단 정렬 차이

변수 두 개면, 대부분의 “빈 공간/정렬 깨짐”을 잡을 수 있습니다.

## 2) DOM 선택자는 텍스트 앵커로 시작

클래스는 바뀌기 쉽기 때문에  
`최신 글` 같은 고정 텍스트를 앵커로 잡고 하위 구조를 찾습니다.

```js
const heading = [...document.querySelectorAll("h2")]
  .find((el) => el.textContent?.trim() === "최신 글");
```

## 3) 계산은 페이지 내부에서 끝낸다

`evaluate`에서 바로 숫자를 계산해 반환하면  
로그/보고용 데이터가 깔끔해집니다.

```js
const gap = Math.max(0, Math.round(cell.bottom - card.bottom));
```

## 4) 성공 기준을 숫자로 못 박는다

프로젝트마다 허용 오차를 정하면 좋습니다.

- 정렬/간격: `<= 8px`
- 콘솔 에러: `0`
- 주요 페이지 3개 이상 스모크 통과

여기까지 정하면 “수정 완료”의 정의가 분명해집니다.

## 5) 해상도는 최소 4구간

저는 보통 아래 4개를 고정합니다.

- 390×844 (모바일)
- 1024×768 (태블릿)
- 1280×800 (데스크톱 기준)
- 1536×960 (와이드)

특히 1024~1280 사이에서 그리드 이슈가 많이 드러납니다.

## 6) 시각 검증과 수치 검증을 함께 저장

스크린샷만 있으면 논쟁이 생기고, 숫자만 있으면 맥락이 부족합니다.  
둘을 같이 남기면 리뷰가 빨라집니다.

## 오늘의 템플릿

```text
페이지: /, /blog, /about
지표: blankGap, deltaBottom
성공기준: gap <= 8, console error 0
해상도: 390/1024/1280/1536
```

레이아웃 QA는 감각이 아니라 데이터로 마무리할수록  
수정 횟수와 피로가 줄어듭니다.
