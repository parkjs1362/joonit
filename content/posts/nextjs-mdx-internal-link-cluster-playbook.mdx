글 하나씩 잘 써도, 독자가 한 페이지만 보고 나가면 누적 효과가 약합니다.  
그래서 저는 글 단위보다 **클러스터 단위**로 묶어서 운영합니다.

## 1) 단발성 글 운영의 한계

단발성 운영에서 자주 생기는 문제는 다음과 같습니다.

- 검색 유입은 있는데 세션당 페이지뷰가 낮음
- 좋은 글이 있어도 아카이브 안에서 묻힘
- 독자에게 "다음 읽을 글"이 자동으로 제시되지 않음

결과적으로 글 품질과 별개로 탐색 경험이 끊깁니다.

## 2) 허브 1 + 위성 3 구조

한 주제에 대해 아래 구조를 기본으로 둡니다.

- 허브 글 1편: 개념/지도/선택 기준
- 위성 글 3편: 실행법/사례/실패 대응

예를 들어 `MDX 운영` 클러스터라면:

1. 허브: MDX 콘텐츠 운영 프레임워크
2. 위성 A: 드래프트 백로그 우선순위
3. 위성 B: 품질 게이트 체크리스트
4. 위성 C: 주간 운영 SLO와 회고

핵심은 허브가 "색인 페이지" 역할을 해야 한다는 점입니다.

## 3) 링크 삽입 위치 규칙

내부 링크는 많이 넣는 것보다 위치가 중요합니다.

- 서론 1개: 이 글이 어느 클러스터에 속하는지 안내
- 본문 중간 1~2개: 맥락상 필요한 상세 글 연결
- 마무리 2개: 다음 액션 기준으로 읽기 순서 제시

실무에서는 "관련 글" 블록보다 **문장 안 인라인 링크**가 클릭률이 더 안정적이었습니다.

## 4) 실행 절차

1. 클러스터 주제 1개를 정하고 허브 글부터 작성합니다.
2. 위성 글 3개의 역할(실행/사례/실패)을 나눕니다.
3. 각 글에 최소 3개의 내부 링크를 배치합니다.
4. 링크 앵커 텍스트를 모호한 표현 대신 문제 해결형으로 씁니다.
5. 발행 후 2주간 클릭 흐름을 보고 링크 위치를 조정합니다.

## 5) 운영 지표

내부 링크 구조의 성과는 아래 지표로 판단합니다.

- 세션당 페이지뷰
- 내부 링크 클릭률(CTR)
- 다음 글 이동률
- 클러스터 단위 평균 체류 시간

글을 잘 쓰는 것과 별개로, **읽히는 동선**을 설계해야 누적 아카이브가 자산이 됩니다.

## 체크리스트

- [ ] 허브 1편과 위성 3편 역할이 분리됐는가
- [ ] 각 글 서론/중간/마무리에 내부 링크가 배치됐는가
- [ ] 앵커 텍스트가 문제 해결형 문장인가
- [ ] 2주 후 지표 리뷰 일정이 캘린더에 있는가

내부 링크는 SEO 보조 수단이 아니라, 독자의 다음 행동을 설계하는 제품 기능에 가깝습니다.
