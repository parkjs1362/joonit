글은 많이 쓰는데 품질 편차가 크다면, 편집 역량보다 **발행 게이트 부재**를 먼저 의심해야 합니다.

Next.js + MDX 블로그에서 실제로 자주 터지는 문제는 비슷합니다.

- frontmatter 누락
- 제목/설명 중복
- 잘못된 카테고리 값
- 이미지 alt 누락
- 링크/OG 메타 불일치

이 글에서는 발행 전에 자동으로 막아야 하는 품질 게이트를 정리합니다.

## 품질 게이트를 나누는 기준

발행 전 검사는 3층으로 나누는 게 운영하기 쉽습니다.

1. **형식 게이트**: 필수 필드 존재 여부
2. **콘텐츠 게이트**: 길이/중복/태그 품질
3. **배포 게이트**: lint/build/라우팅 검증

한 번에 모든 걸 강제하면 팀이 우회합니다.  
가벼운 규칙부터 고정해야 지속됩니다.

## 필수 frontmatter 규칙

최소 필드는 아래 5개를 고정합니다.

- `title`
- `description`
- `date`
- `category`
- `tags`

`category`는 허용 값 목록과 비교해야 오타를 막을 수 있습니다.  
예: 개발/역사/일상/여행.

## 중복과 약한 설명을 막는 방법

`description`이 제목 복붙이면 카드 클릭률이 떨어집니다.  
설명은 “이 글에서 얻는 결과”를 한 문장으로 적는 규칙이 좋습니다.

또한 아래 조건을 간단히 검사합니다.

- 제목 12자 미만이면 경고
- 설명 40자 미만이면 경고
- 태그 1개 이하이면 경고

경고는 빌드 실패가 아니라 수정 유도로 시작하는 편이 현실적입니다.

## 발행 직전 스모크 항목

기술적으로는 아래 4가지만 지켜도 사고가 크게 줄어듭니다.

- `npm run lint`
- `npm run build`
- 신규 슬러그 2개 직접 접속 확인
- OG/Twitter 메타 1건 확인

정적 사이트라도 메타와 라우팅은 수동 1회가 필요합니다.

## 운영 팁: 글쓰기와 검증을 분리한다

작성자는 내용에 집중하고, 발행자는 게이트를 통과시키는 방식이 효율적입니다.  
1인 운영이라도 “작성 시간”과 “검증 시간”을 나누면 오류가 줄어듭니다.

## 체크리스트

- [ ] frontmatter 필수 필드가 모두 있는가
- [ ] 카테고리가 허용 값에 포함되는가
- [ ] 제목/설명이 서로 다른 정보를 담는가
- [ ] lint/build가 모두 통과하는가
