QA를 강화하자고 하면 보통 테스트 항목이 먼저 늘어납니다.  
하지만 소규모 팀에서는 항목 확대보다 **조합 최소화**가 먼저입니다.

핵심은 “중요한 오류를 가장 싼 비용으로 빨리 잡는 매트릭스”를 만드는 것입니다.

## 왜 매트릭스가 필요한가

프론트엔드 이슈는 대체로 3가지 축에서 발생합니다.

- 뷰포트(모바일/데스크톱)
- 테마(라이트/다크)
- 경로(홈/목록/상세/정책)

세 축을 무작정 곱하면 운영이 불가능해집니다.  
그래서 **고정 조합**을 먼저 정해야 합니다.

## 추천 기본 조합(8개)

- 390x844, 라이트: `/`, `/blog`
- 390x844, 다크: `/`, `/blog`
- 1280x800, 라이트: `/`, `/blog`, `/blog/<slug>`
- 1280x800, 다크: `/`, `/blog`, `/blog/<slug>`

정책 페이지(`/about`, `/privacy`)는 주 1회 확장 점검으로 분리합니다.

## 성공 기준을 숫자로 둔다

단순 “문제 없음”은 기록에 남지 않습니다.  
아래처럼 수치 기준을 둡니다.

- 콘솔 `error`: 0
- 404 응답: 0
- 카드 높이 편차: 8px 이하
- 가로 스크롤: 0

수치가 있으면 배포 판단이 빨라집니다.

## 자동화와 수동 확인의 경계

전부 자동화하려 하면 오히려 유지보수 비용이 커집니다.  
소규모 팀은 다음처럼 나누는 게 현실적입니다.

- 자동: 라우팅, 콘솔 에러, 레이아웃 수치
- 수동: 텍스트 가독성, 색 대비 체감, 콘텐츠 오탈자

자동화는 기계가 잘하는 일만 맡겨야 오래 갑니다.

## 주간 운영 루틴

- 배포 전: 핵심 8조합 스모크
- 주간 1회: 확장 페이지 점검
- 월간 1회: 테스트 항목 정리(사용 안 하는 항목 삭제)

추가보다 정리가 더 중요합니다.

## 체크리스트

- [ ] 핵심 8조합이 문서화되어 있는가
- [ ] 콘솔/404/편차 기준이 숫자로 정의돼 있는가
- [ ] 자동과 수동 경계가 명확한가
- [ ] 배포 전 점검이 15분 안에 끝나는가

## Joonit의 관점

QA는 완벽을 증명하는 과정이 아니라 **배포 리스크를 통제하는 과정**입니다.  
저는 테스트 개수보다 “배포 직전에 믿을 수 있는 핵심 조합”이 있는지를 더 중요하게 봅니다.
