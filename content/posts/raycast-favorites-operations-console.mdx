런처는 보통 "앱 여는 도구"로 끝납니다.  
하지만 운영 자동화가 늘어난 환경에서는 런처가 사실상 콘솔 역할을 해야 합니다.

저는 Raycast를 쓸 때 즐겨찾기 순서를 업무 흐름 순서와 맞춥니다.  
이 작은 변경만으로 점검 누락이 크게 줄었습니다.

## 1) Favorites는 기능순이 아니라 시간순으로 배치

기능별 분류는 예쁘지만 실행 동선은 느립니다.  
아래처럼 시간 흐름 기준으로 배치하는 편이 훨씬 빠릅니다.

1. System Daily Check
2. Dashboard
3. Health Check
4. Nightly Cleanup Report
5. Nightly Cleanup Apply

순서 자체가 운영 프로토콜이 됩니다.

## 2) 파괴적 명령은 연속 배치하지 않는다

실수는 비슷한 위치의 명령에서 자주 발생합니다.  
Report와 Apply를 붙여두더라도 시각적으로 충분히 구분해야 합니다.

- 아이콘/이름에 동사 명확히 표기
- Apply는 접미사로 강조
- 실행 전 확인 문구 일관화

작은 라벨 규칙이 사고를 막습니다.

## 3) 일일 루틴과 주간 루틴을 분리한다

모든 명령을 매일 실행할 필요는 없습니다.

- 일일: Daily Check, Dashboard
- 주간: Health Check, 로그 분석

이 분리가 없으면 좋은 도구도 금방 피로의 원인이 됩니다.

## 4) 콘솔 품질은 "실행률"로 본다

정교한 설정보다 중요한 지표는 실행률입니다.  
매일 1회 이상 실행되는지, 주간 리뷰가 실제로 도는지 확인해야 합니다.

실행률이 떨어지면 기능 추가보다 동선을 먼저 줄여야 합니다.

## 체크리스트

- Favorites 순서가 하루 작업 흐름과 일치하는가
- Report/Apply 명령이 명확히 구분되는가
- 일일 루틴과 주간 루틴 명령을 분리했는가
- 최근 7일 실행률을 확인하고 있는가

## 오늘의 메모

좋은 런처는 빠른 검색보다  
좋은 순서 설계에서 시작됩니다.
