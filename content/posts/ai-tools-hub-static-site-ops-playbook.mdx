AI 도구 허브는 초기 제작보다 **지속 업데이트**가 더 어렵습니다.  
정적 HTML/CSS/JS 구조는 단순하지만, 운영 규칙이 없으면 금방 유지보수 피로가 커집니다.

## 1) 구조 단순화는 유지보수 비용 절감에 직접적이다

외부 의존성이 적을수록 장애 포인트가 줄어듭니다.  
정적 사이트는 배포 경로가 단순해 회귀를 빠르게 파악하기 좋습니다.

대신 코드가 커질수록 기능 추가 기준을 엄격하게 관리해야 합니다.

## 2) 콘텐츠 갱신 루프를 먼저 설계한다

허브 사이트의 핵심은 UI보다 데이터 갱신 루프입니다.

- 도구 목록 업데이트 주기
- 뉴스 소스 검증 주기
- 죽은 링크 정리 주기

이 세 주기를 고정하면 “관리되는 허브”가 됩니다.

## 3) 북마크/도구 데이터는 카테고리 기준으로 분리한다

단일 파일 구조라도 데이터 섹션을 카테고리별로 명확히 분리하면 충돌이 줄어듭니다.  
운영 중에는 새 링크 추가보다 **중복/품질 정리**가 더 중요합니다.

## 4) PWA는 기능보다 신뢰성 보강 수단으로 쓴다

PWA 적용의 목적은 화려한 기능이 아니라 접속 안정성과 재방문 경험 개선입니다.  
Service Worker 정책은 단순하고 예측 가능하게 유지해야 운영이 쉽습니다.

## 체크리스트

- [ ] 업데이트 주기(도구/뉴스/링크)가 문서화돼 있는가
- [ ] 데이터 섹션이 카테고리별로 분리돼 있는가
- [ ] PWA 캐시 정책이 단순하게 유지되는가
- [ ] 죽은 링크 점검 루프가 있는가

## Joonit의 관점

허브형 사이트는 디자인보다 **갱신 시스템**이 경쟁력이라고 봅니다.  
저는 새 기능보다 먼저 운영 루프가 유지되는지 확인합니다.
