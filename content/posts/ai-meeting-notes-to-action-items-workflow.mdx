회의가 끝나면 요약은 남는데, 실제 액션은 자주 사라집니다.  
문제는 기록이 부족해서가 아니라 **실행 단위로 변환되지 않기 때문**입니다.

이 글은 회의 내용을 AI로 정리한 뒤, 바로 실행 가능한 항목으로 연결하는 운영 절차를 다룹니다.

## 왜 요약만으로는 실행이 안 붙는가

요약 문장은 보통 “무엇을 논의했는가”를 담습니다.  
하지만 실행에는 최소한 아래 4가지가 필요합니다.

- 누가(owner)
- 언제까지(due date)
- 어떤 완료 조건(done definition)
- 어떤 선행 조건(dependency)

이 중 하나라도 비면, 업무는 다시 메신저 대화로 흩어집니다.

## 5단계 변환 프로토콜

### 1) 회의 직후 10분 안에 원문을 정리한다

회의 직후가 가장 정확합니다.  
발언 의도와 맥락이 살아 있을 때 원문을 정리해야 AI 추출 정확도가 올라갑니다.

### 2) AI에게 “결정/보류/액션” 3분류를 강제한다

프롬프트에서 분류를 명시해야 합니다.

- 결정사항(Decision)
- 보류사항(Parking Lot)
- 실행항목(Action Item)

이 분류가 없으면 액션과 의견이 섞입니다.

### 3) 액션 항목은 반드시 1문장 1행동으로 쪼갠다

“배포 안정성 개선” 같은 문장은 과제명이 아니라 방향입니다.  
예: “배포 전 체크리스트에 `npm run build`와 OG 메타 확인 항목 추가”처럼 구체화합니다.

### 4) 완료 조건을 체크리스트로 붙인다

완료 조건이 모호하면 진행률이 허수로 부풀어집니다.

- 재현 경로 1개
- 검증 스크린샷 1장
- 회귀 영향 1줄

### 5) 24시간 내 재확인 루프를 돌린다

회의 당일에 생성한 액션은 다음 날 1회 재확인해야 정확해집니다.  
팀의 우선순위가 바뀌면 이때 바로 반영합니다.

## 실패를 줄이는 운영 규칙

- 회의마다 템플릿을 바꾸지 않는다
- Owner 없는 항목은 즉시 보류로 보낸다
- Due date 없는 항목은 “이번 주 검토”처럼 범위를 지정한다
- 실행 항목은 최대 7개로 제한한다

항목 수를 줄여야 완료율이 올라갑니다.

## 바로 적용할 템플릿

```md
[Decision]
- ...

[Action Items]
- [ ] 작업명 / Owner / Due / Done 조건

[Parking Lot]
- ...
```

## 체크리스트

- [ ] 액션 항목에 Owner가 모두 지정됐는가
- [ ] 각 항목에 완료 조건이 1개 이상 있는가
- [ ] 보류 항목과 실행 항목이 분리됐는가
- [ ] 24시간 내 재확인 일정을 잡았는가
