여러 브라우저를 동시에 쓰면 북마크 동기화는 필연적으로 충돌합니다.  
문제를 줄이는 핵심은 알고리즘보다 **동기화 순서와 실패 처리 규칙**입니다.

## 1) 중앙 저장소를 단일 기준으로 둔다

브라우저끼리 직접 동기화하면 충돌 원인 추적이 어려워집니다.  
중앙 JSON 저장소를 기준으로 읽기/쓰기 경로를 단순화해야 운영이 쉬워집니다.

## 2) 브라우저 상태를 먼저 확인한다

특정 브라우저는 실행 중일 때 북마크 파일을 덮어쓰면 종료 시 원래 상태로 되돌아갈 수 있습니다.  
그래서 동기화 전에 실행 중 여부를 먼저 확인하는 단계가 필요합니다.

## 3) 동기화는 단계적으로 분리한다

- 소스 데이터 검증
- 브라우저별 반영
- 반영 결과 기록

이 3단계를 분리하면 실패 지점을 즉시 찾을 수 있습니다.

## 4) 실패를 히스토리로 남긴다

동기화 도구에서 가장 중요한 것은 마지막 성공 시각과 실패 원인 기록입니다.  
브라우저별 상태 기록이 있어야 “어느 쪽이 최신인지”를 판단할 수 있습니다.

## 5) 자동 스케줄은 보수적으로 시작한다

시간 단위 자동 동기화는 편하지만, 초기에는 수동 실행 기반으로 검증 기간을 두는 편이 안전합니다.  
충돌 패턴이 안정된 뒤 주기 자동화를 붙이는 방식이 재작업을 줄입니다.

## 체크리스트

- [ ] 중앙 저장소가 단일 기준으로 유지되는가
- [ ] 실행 중 브라우저 처리 규칙이 있는가
- [ ] 브라우저별 동기화 결과가 기록되는가
- [ ] 자동 스케줄 전 수동 검증 기간을 거쳤는가

## Joonit의 관점

동기화 도구는 기능보다 **복구 가능성**이 더 중요합니다.  
저는 새 브라우저를 추가할 때 항상 성공 경로보다 실패 복구 경로부터 먼저 확인합니다.
