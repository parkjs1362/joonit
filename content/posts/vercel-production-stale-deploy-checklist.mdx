코드를 수정하고 `main`에 푸시했는데 화면이 그대로라면,  
대부분은 코드 문제가 아니라 **배포 경로 불일치**입니다.

아래 8개를 순서대로 보면, 원인을 꽤 빠르게 좁힐 수 있습니다.

## 1) 리포지토리의 실제 HEAD 확인

가장 먼저 로컬과 원격이 같은 커밋인지 봅니다.

- `git status -sb`
- `git log --oneline -3`

여기서 커밋이 다르면, 배포 이전에 Git 상태부터 정리해야 합니다.

## 2) Production Branch 설정 확인

Vercel 프로젝트에서 Production Branch가 `main`인지 반드시 확인합니다.  
간혹 `master`나 다른 브랜치가 남아 있어 푸시해도 반영이 안 됩니다.

## 3) 연결된 GitHub 저장소 확인

유사한 이름의 프로젝트가 여러 개면  
다른 저장소에 연결된 Vercel 프로젝트를 보고 있을 수 있습니다.

운영 중인 도메인과 실제 프로젝트 연결을 한 번 더 맞춰야 합니다.

## 4) 배포는 성공했는지, 어떤 커밋으로 성공했는지 확인

“배포 성공”과 “내 커밋이 배포됨”은 다릅니다.

- 배포 로그의 commit SHA가 내가 푸시한 SHA와 같은지
- 배포 시각이 최근 푸시 이후인지

두 조건이 맞아야 진짜 반영입니다.

## 5) 마커로 화면 반영 여부 즉시 확인

홈 루트에 `data-layout-version="home-v3"` 같은 마커를 두면  
배포 확인 시간이 크게 줄어듭니다.

브라우저에서 DOM을 한 번 보면 “새 코드/구 코드”를 즉시 구분할 수 있습니다.

## 6) 브라우저 캐시와 서비스워커 영향 분리

크롬 강력 새로고침만으로 해결되지 않는 경우가 있습니다.

- 시크릿 창 확인
- 다른 브라우저 확인
- 등록된 서비스워커가 있는지 확인

캐시 문제를 분리하면, 서버 문제인지 클라이언트 문제인지 나뉩니다.

## 7) 수치로 반영 확인 (레이아웃)

“보기에 비슷하다” 대신 숫자를 남깁니다.

- 예: `blankGap = Math.max(0, firstCell.bottom - firstLink.bottom)`
- 성공 기준: `blankGap <= 8`

이렇게 두면 QA 기준이 팀 내에서 일관됩니다.

## 8) 마지막으로 도메인 라우팅 확인

커스텀 도메인이 다른 프로젝트를 보고 있는 경우가 있습니다.  
`*.vercel.app`와 커스텀 도메인을 둘 다 확인해 불일치를 제거합니다.

## 오늘의 체크리스트

- 브랜치/저장소/커밋 SHA를 먼저 맞춘다
- DOM 마커로 반영 여부를 즉시 확인한다
- 시각 QA는 수치 QA와 함께 본다

배포 문제는 보통 “복잡한 버그”보다 “경로 불일치”인 경우가 많습니다.
