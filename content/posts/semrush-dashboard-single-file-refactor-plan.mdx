대시보드 프로젝트를 운영하다 보면 App 파일이 커지는 순간이 옵니다.  
많은 팀이 즉시 분할 리팩토링을 선택하지만, 운영 단계에서는 **시점 판단**이 더 중요합니다.

## 1) 큰 파일 자체가 문제는 아니다

단일 파일도 배포 안정성과 기능 변경 속도가 유지되면 당장 분할할 이유는 없습니다.  
문제는 파일 크기가 아니라 아래 신호입니다.

- 변경 충돌 빈도 증가
- 버그 수정 리드타임 증가
- 신규 기능 추가 시 회귀 빈도 증가

이 지표가 올라갈 때 구조 분할을 시작하는 편이 비용 대비 효과가 큽니다.

## 2) 분할은 데이터 파이프라인부터

대시보드에서는 UI보다 데이터 변환 체인이 핵심 복잡도입니다.  
필터링/정렬/페이지네이션 경로를 먼저 분리하면 이후 컴포넌트 분리가 쉬워집니다.

## 3) 도움말/문서와 실제 기능을 계속 맞춘다

운영이 길어질수록 UI 도움말과 실제 동작이 어긋나기 쉽습니다.  
문서 불일치는 사용자 혼란을 만들고 지원 비용을 늘립니다.

## 4) 성능 개선은 측정 후 최소 변경으로

무작정 리팩토링보다 병목 지점을 먼저 측정해야 합니다.  
번들 분할, 캐시, 메모이제이션은 하나씩 적용하고 효과를 수치로 확인해야 합니다.

## 체크리스트

- [ ] 리팩토링 시작 기준 지표가 정의돼 있는가
- [ ] 데이터 파이프라인이 우선 분리돼 있는가
- [ ] 도움말과 실제 기능이 일치하는가
- [ ] 성능 개선이 측정 기반으로 진행되는가

## Joonit의 관점

리팩토링은 미덕이 아니라 **투자 결정**이라고 봅니다.  
저는 구조 개편 전에 운영 지표가 실제로 악화됐는지부터 먼저 확인합니다.
